On this page

Sandbox ​

tnk uses Lima for sandbox virtualization. Each project gets its own Linux VM via Apple's VZ hypervisor.

Lima VMs ​

Each project gets its own Lima VM. Lima runs lightweight Linux VMs via Apple's VZ hypervisor on Apple Silicon.

VM image ​

Each VM boots from Lima's default template:ubuntu which ships with Ubuntu and containerd/nerdctl.

Instance naming ​

VM instance names follow the convention:

~/code/homepage → tnk-homepage
~/code/tnk    → tnk-tnk

The VM is created on first use and persists across sessions. Manage with tnk sandbox commands.

Networking ​

The tnk-services VM runs the MCP bridge and SearXNG search tooling in an isolated sandbox — separate from project workspaces and the host. Per-project sandboxes reach these services via the runtime environment contract. The inference engine on the host is reachable from within the sandbox at the configured bind address.

Mount configuration ​

Each sandbox mounts the project workspace into /workspace inside the VM. Only the project directory mounts by default.

Provisioning ​

tnk copies the profile provision script and shared provisioning library into /tmp inside the VM, then executes the script as the tnk user.

Shared concepts ​

Content-addressed provisioning ​

tnk computes a double SHA-256 hash of every file in the provision script directory and the shared library. The resulting digest becomes TNK_SPECS_REV, injected as an environment variable into the sandbox.

Adding, removing, or editing any file in the provision system triggers a new provision run.

Config revision tracking ​

Services track configuration revisions. When tnk detects that a service instance exists but its config revision does not match the current version, it recreates the instance with updated settings.

Baseline setup ​

Every sandbox gets a baseline setup before the profile script runs:

  • Required tooling installs (bash, curl, git, nodejs, npm)
  • The tnk user exists with a matching UID/GID to the host user when possible
  • Home directory setup inside the sandbox (~/.bashrc, ~/.profile)

UID/GID matching prevents file ownership drift on bind mounts. If the host user lacks a matching UID in the sandbox, tnk falls back to UID 1000.

Network posture ​

Profiles declare a network policy. The default is full internet access. Profiles can request network restricted for constrained execution. tnk applies the policy through the sandbox backend's network API.

Audit log format ​

--audit-log PATH writes NDJSON to the specified file. Each line is a complete JSON object. Example structure:

json
{"event":"session_start","ts":"2026-01-15T10:23:41Z","project":"tnk","tty":true,"runtime":"llama","inference_url":"http://host:8080"}
{"event":"command","ts":"2026-01-15T10:23:42Z","argv":["hermes","agent","--run"]}
{"event":"session_exit","ts":"2026-01-15T10:31:07Z","exit_code":0,"duration":446}
